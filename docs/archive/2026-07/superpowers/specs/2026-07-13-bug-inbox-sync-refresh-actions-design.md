# Bug 收件箱同步与刷新操作设计

## 背景

Bug 工单页同时存在“同步我的 Bug”和“刷新”两个动作。前者会访问当前 Bug 平台、拉取指派工单并写入本地工单库；后者只重新读取本地工单列表。当前文案和视觉层级没有表达这一数据边界，用户容易认为两个动作重复。

## 目标

- 明确区分“外部平台同步”和“本地列表刷新”。
- 保留本地刷新能力，使 webhook 或其他平台写入的工单可以立即显示。
- 优化刷新按钮的图标、层级、加载反馈、键盘可访问性和移动端触控尺寸。
- 不改变后端接口、持久化模型和现有同步流程。

## 交互设计

### 从平台同步

- 将“同步我的 Bug”改为“从平台同步”。
- 使用内联 SVG 同步图标，不使用 emoji 或字符图标。
- 执行期间禁用按钮并显示“同步中…”。
- 继续调用 `syncBugPlatform`；成功后仍刷新本地列表并保留现有成功/错误反馈。

### 刷新本地列表

- 将“刷新”改为“刷新列表”。
- 按钮位于 Bug 收件箱标题区域右侧，使用紧凑描边的次要操作样式。
- 使用内联 SVG 刷新图标；加载时图标旋转，同时按钮禁用并显示“刷新中…”。
- 只调用本地 `listBugs` 读取流程，不触发 `syncBugPlatform`。
- 保留 44px 移动端触控目标、可见焦点和 `prefers-reduced-motion` 支持。

## 视觉层级

- “从平台同步”属于平台配置中的外部数据动作，沿用配置区次要按钮。
- “刷新列表”属于收件箱局部工具，只影响当前列表，视觉权重低于保存配置和故障闭环主操作。
- 两个动作使用不同文案和所在区域表达作用域，不依赖颜色区分。

## 测试

- 验证两个按钮的新文案、SVG 图标和独立操作标识。
- 验证点击“刷新列表”只重新调用 `listBugs`，不会调用 `syncBugPlatform`。
- 验证点击“从平台同步”调用 `syncBugPlatform`，同步完成后重新读取列表。
- 验证加载期间按钮禁用、加载文案与旋转图标状态。
- 运行 Bug 工单页聚焦测试、全量 Web 测试、生产构建和 lint。

## 非目标

- 不合并两个操作。
- 不新增自动刷新、轮询或新的同步策略。
- 不修改 Bug 平台后端接口与数据库结构。
