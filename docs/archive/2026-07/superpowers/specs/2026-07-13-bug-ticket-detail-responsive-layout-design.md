# Bug 工单详情响应式布局优化设计

日期：2026-07-13

## 背景

Bug 工单页右侧详情区在真实中文工单和长附件名称下存在明显布局问题：Markdown 有序列表的序号落到内容卡片之外，长文件名挤压“预览”提示，标题、状态标签、字段卡片和底部操作只在极窄屏才重排，中等宽度下仍可能拥挤。

本次只优化共享组件 `BugTicketDetail` 的完整详情模式，不调整左侧工单列表、刷新按钮、平台配置区或故障闭环业务行为。

## 方案选择

采用组件级自适应重排：在保留现有视觉语言和 DOM 语义的基础上，为富文本、标题、字段、附件和底部操作建立明确的内容约束与容器断点。

未采用以下方案：

- 只修复列表缩进：不能解决附件文件名、状态标签和操作按钮在中等宽度下的拥挤。
- 重做整个 Bug 工单页面：会扩大到左侧列表和平台配置，超出当前问题范围。

## 详细设计

### 富文本内容

- `.ticket-markdown` 使用 `box-sizing: border-box` 和 `min-width: 0`，保证 padding 和边框计入可用宽度。
- `ol`、`ul` 使用明确的左侧内边距，列表标记采用 `outside`，但标记和正文整体均处于卡片内容区内。
- 段落、列表项、标题、链接和行内代码允许安全断词，连续英文、URL 和中文混排不得撑宽父容器。
- `pre` 和 `table` 在内容块内部横向滚动，不让页面或详情面板横向滚动；普通文本仍自然换行。
- 列表和段落使用一致的垂直间距与 1.6 左右行高，保持长复现步骤可扫描。

### 标题、状态和字段

- 宽容器下标题和状态标签左右排列。
- 详情容器宽度不足时，标题与状态改为上下排列，状态标签左对齐；断点依据组件可用宽度，不依赖整个浏览器宽度。
- 完整字段宽容器下保持两列，容器不足时切换单列。
- 标题、字段值和标签均允许换行，任何单项内容不得扩大网格轨道。

### 附件卡片

- 保留整个附件卡片作为唯一可点击按钮和至少 44px 的触控区域。
- 图标、文件信息和“预览”操作使用稳定三列；文件信息列必须 `minmax(0, 1fr)`。
- 文件名最多显示两行并省略视觉溢出，按钮通过 `title` 和包含完整文件名的 `aria-label` 提供完整信息。
- “预览”作为固定宽度的操作标识，不因文件名长度移动或换行。
- 宽容器最多两列；中窄容器切换为单列，避免两个附件卡片互相挤压。

### 底部操作

- 宽容器下“进入故障闭环”保持右对齐。
- 窄容器下操作区占满宽度，按钮为整行宽度，保持可见焦点态和至少 44px 高度。

## 可访问性与响应式

- 附件按钮保留原生 `button` 语义、键盘操作和可见焦点。
- 省略显示的附件名不得丢失完整可访问名称。
- 不通过颜色单独表达可点击状态；保留文本“预览”。
- 在 375、768、1024、1440 像素目标视口下，详情内容不得造成页面级横向溢出。
- 尊重现有 `prefers-reduced-motion` 设置，不增加布局位移动画。

## 测试策略

- 使用包含有序列表、超长连续文本、长附件文件名和多个状态标签的 fixture。
- 断言富文本列表、代码块和表格具有专用的内容约束标识。
- 断言附件按钮具有完整 `aria-label`、`title` 和稳定的文件信息/操作结构。
- 断言组件声明容器自适应契约和 375、768、1024、1440 四档支持。
- 保留现有 Markdown 消毒、附件点击、进入故障闭环和空状态测试。

## 验收标准

- 复现步骤的列表序号和项目符号完全位于内容卡片内。
- 长文本、URL、代码块、表格和字段值不会撑破右侧详情区。
- 长附件文件名不会挤压或移动“预览”操作，完整名称仍可访问。
- 中窄宽度下标题、字段、附件和底部按钮按设计重排。
- 左侧工单列表、平台配置和故障闭环数据流没有行为变化。
