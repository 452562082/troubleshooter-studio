# 故障闭环机器人操作区设计

日期：2026-07-14

## 背景

故障闭环页当前把机器人选择放在顶部右栏，但只在“从未创建 Case”或“全部 Case 已结束”时，才在中间 Bug 摘要栏显示启动按钮。Bug 已有活动 Case 时，顶部启动按钮消失，唯一操作入口落在三栏选择区下方的 Case 生命周期中。顶部区域最高可占约 58vh，用户完成机器人选择后容易误以为没有入口。

现有重置语义还会强制沿用旧 Case 的机器人和环境。这与新的交互预期冲突：上方机器人选择应当是本次开启或重新开始动作的明确输入。

## 目标

- 在“选择排障机器人”面板内集中展示开启、进入和重新开始操作。
- 用户完成机器人选择后，无需寻找首屏外入口即可采取下一步操作。
- 活动 Case 同时提供“进入故障闭环”和“重新开始故障闭环”。
- 重新开始使用上方当前选中的机器人和环境，而不是旧 Case 的绑定。
- 旧 Case、证据、事件和外部副作用保持不可变且可追溯。
- 保持现有 Case 唯一性、版本控制、幂等和 Agent 调度护栏。

## 非目标

- 不把重置变成无确认的一键操作。
- 不回滚旧 Case 已产生的提交、推送、合并或人工部署。
- 不改变验证、排障、修复、合并、部署和回归的状态机阶段。
- 不在本次改动中重排 Case 生命周期、时间线或证据详情布局。
- 不允许 Cursor 通过 Studio 后台直启；沿用现有 Codex、Claude Code 和 OpenClaw 支持范围。

## 方案

采用机器人面板内的统一操作区。中间 Bug 摘要栏不再承载独立启动卡，避免同一动作分散在两个区域。

### 状态与操作

| 当前 Bug 状态 | 主操作 | 次操作 | 行为 |
|---|---|---|---|
| 没有任何 Case | 开启故障闭环 | 无 | 使用当前机器人和环境创建 Case 并启动验证 |
| 存在未结束 Case | 进入故障闭环 | 重新开始故障闭环 | 进入定位当前 Case；重新开始先确认，再归档旧 Case 并用当前选择创建接替 Case |
| 只有已结束 Case | 进入故障闭环 | 重新开始故障闭环 | 进入定位最新历史 Case；重新开始使用当前选择创建新一轮 |

终态继续沿用 `fixed_verified`、`legacy_archived` 和 `reset_archived`。存在活动 Case 时，活动 Case 优先于历史 Case 作为“进入”和“重新开始”的上下文。

### 操作区布局

操作区位于 `BugBotPicker` 下方，与机器人选项保持同一面板和阅读顺序：

1. 一行短状态说明，例如“已有进行中的 Case · 等待证据”或“尚未建立 Case”。
2. 主按钮使用现有 primary 样式，最小点击高度 44px。
3. “重新开始故障闭环”使用次要危险样式，不与主按钮争夺视觉层级。
4. 375px 等窄屏下按钮纵向排列并占满可用宽度；较宽布局下可并排。

机器人匹配中、未选机器人、选中不支持后台直启的 target，或命令正在执行时，开启和重新开始按钮禁用，并在操作区附近显示原因。“进入故障闭环”只依赖可展示的 Case，不因机器人匹配失败而禁用。

### 进入故障闭环

“进入故障闭环”不产生后端写操作。它将：

1. 确保当前活动 Case，或没有活动 Case 时的最新历史 Case，已被选中并加载。
2. 平滑滚动到 `BugCaseLifecycle` 的当前操作卡。
3. 在滚动完成后把键盘焦点放到当前阶段主按钮；若 Case 已终止且没有主按钮，则聚焦 Case 标题区域。
4. 在 `prefers-reduced-motion: reduce` 下使用即时滚动。

Case 详情仍在加载时，入口保持可用并先定位到加载状态；详情完成后再执行一次焦点定位。加载失败沿用页面 live error 和刷新入口。

### 开启故障闭环

“开启故障闭环”复用现有 `StartIncidentCase`：

- 输入使用当前 Bug、当前选中的 `bot_key` 和该匹配项的环境。
- 前端继续生成稳定的单次请求 Case ID 与幂等键。
- 后端发现同一 Bug 已存在活动 Case 时返回已有 Case；前端切换为“进入故障闭环”语义并提示“已打开现有闭环”。
- 成功后自动定位新 Case 的生命周期区域。

### 重新开始故障闭环

“重新开始故障闭环”始终先打开确认框，不直接执行命令。确认框显示：

- 当前 Bug、旧 Case、状态、阶段和 attempt。
- 旧 Case 绑定的机器人与环境。
- 接替 Case 将使用的当前选中机器人与环境。
- 旧 Agent 会被停止。
- 旧证据和审计保留，已发生的外部副作用不会回滚。

确认框继续默认聚焦取消按钮、圈定焦点、支持 Escape 关闭并恢复触发按钮焦点。提交期间关闭和重复确认均禁用。

## 重置命令语义调整

本设计有意取代 `2026-07-13-bug-inbox-incident-workbench-split-design.md` 中“重置沿用环境和所选机器人”的约束。旧 Case 继续保持其原绑定；接替 Case 改用请求中明确指定的新机器人和新环境。

### 当前选择的环境解析

上方选择器展示的环境必须与写入新 Case 的环境一致。`MatchBugBots` 根据 Bug 的 `platform_id` 读取对应平台配置，并把同一 `bot_key` 的 `bot_mappings[].env` 写入返回的 `BotRef.Env`。环境解析顺序固定为：

1. 当前机器人对应的 `bot_mappings[].env`。
2. Bug 同步时保存的 `bot_env`，用于兼容旧的单环境平台配置。
3. Bug 的目标 `env`。

最终环境回填到匹配结果并显示为“环境 <id>”。开启和重新开始都使用该 `BotRef.Env`，内部统一通过一个环境解析 helper 保持 Create、Reset 与恢复逻辑一致。不得在前端显示 `test`，但后端又按 Bug 的其它字段写入不同环境。

### 输入与校验

桌面 binding 和内部命令继续接收 `bot_key`，但其含义改为接替 Case 的目标机器人。binding 根据该 key 加载当前机器人映射，不再要求它等于旧 Case 的 `selected_bot_key`。

重置前必须校验：

- 旧 Case、Bug、版本和幂等键有效。
- 新机器人存在且支持 Studio 后台启动。
- 新环境使用上方匹配结果已解析出的 `BotRef.Env`。
- 新环境非空；环境缺失时在确认前阻止提交并显示可恢复错误。
- 当前 attempt 可以属于旧机器人；该事实用于取消旧 Agent，不限制接替 Case 的机器人。

### 原子持久化

`CaseReset` 请求增加接替 Case 的机器人、target 和环境快照，并进入请求 fingerprint。SQLite 重置事务继续原子完成：

1. 归档旧 Case 为 `reset_archived`，保留其原机器人和环境。
2. 写入双向重置关系和审计事件。
3. 创建 `pending_validation` 接替 Case，写入新机器人 key 和新环境。
4. 提交后使用新机器人 target 调度验证 Agent。

重放相同幂等请求返回第一次创建的接替 Case。相同幂等键但机器人或环境不同必须返回幂等冲突。旧 Agent 的晚到回调继续由 Case version、attempt status 和 claim 检查阻断。

历史终态重新开始不需要先归档旧 Case。它复用创建新一轮能力，但使用当前选择的机器人和环境，并保留历史 Case只读。

## 错误与竞态

- 机器人选择在确认框打开后变化：确认框使用打开瞬间的机器人和环境快照；用户关闭后重新打开才能采用新选择。
- Case 在确认期间变化：提交返回版本冲突，关闭 pending，刷新最新 Case 并要求重新确认。
- 重置事务成功但新 Agent 调度失败：保留接替 Case，展示现有结构化 warning，并允许从验证阶段重试。
- 机器人匹配失败：仍可进入和浏览已有 Case，但不能开启或重新开始。
- 点击开启时后端返回活动 Case：不创建重复 attempt，切换并定位该 Case。
- 快速重复点击：前端 pending 与 `runOnce` 阻止重复提交，后端幂等键和事务唯一性继续兜底。

## 可访问性与响应式

- 所有按钮保留可见焦点态和至少 44×44px 点击区域。
- 危险操作不只依赖颜色，按钮文本明确包含“重新开始”。
- 操作状态与错误使用 `role="status"` 或现有 live region。
- 程序化聚焦只在用户触发“进入”后发生，不在后台刷新时抢焦点。
- 375、768、1024、1440px 下不得出现页面级横向滚动。
- 小屏按钮纵向排列，DOM 顺序保持“进入/开启”在前、“重新开始”在后。

## 测试策略

### Web

- 无 Case 时，机器人面板显示“开启故障闭环”，中间摘要不再显示旧启动卡。
- 活动 Case 和仅终态 Case 均同时显示“进入故障闭环”和“重新开始故障闭环”。
- “进入”选择正确 Case、滚动生命周期区域并按可用动作聚焦。
- reduced-motion 下使用即时滚动。
- 开启成功或后端返回现有 Case后自动定位对应 Case。
- 上方机器人匹配结果显示平台中该机器人的映射环境，且开启、确认框和接替 Case 使用同一环境。
- 重新开始确认框快照包含旧绑定与当前新绑定；选择变化不污染已打开确认框。
- 未选机器人、不支持的 target、匹配失败和 pending 状态正确禁用写操作，但不禁用进入。
- 四个目标 viewport 下操作区无溢出、按钮顺序稳定。

### Go / SQLite

- 重置允许旧机器人 A 切换到新机器人 B，并用 B 的环境创建接替 Case。
- 旧 Case 仍保存 A 和旧环境，接替 Case 保存 B 和新环境。
- 运行中旧 attempt 仍按 A 的 runner 身份取消，新验证按 B 调度。
- 重置到缺失环境或不支持 target 的机器人被拒绝，事务无部分写入。
- 相同请求幂等重放只创建一个接替 Case 和一个验证 attempt。
- 同一幂等键更换机器人或环境返回冲突。
- SQLite reopen 后新旧绑定、重置关系和审计事件保持一致。
- 版本冲突、事务失败和晚到旧回调继续满足既有恢复测试。

## 验收标准

- 用户选择机器人后，首屏同一面板内始终能看到与当前 Case 状态匹配的下一步操作。
- 活动 Case同时显示“进入故障闭环”和“重新开始故障闭环”。
- “进入”无需后端写操作即可定位并聚焦当前 Case。
- “重新开始”经确认后，用当前选中的机器人和环境创建接替 Case。
- 旧 Case 的机器人、环境、证据和审计不被改写。
- 重复、并发、失败和应用重启不会创建重复 Case、attempt 或 Agent 副作用。
