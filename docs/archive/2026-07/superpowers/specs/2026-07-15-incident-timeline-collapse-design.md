# 故障闭环过程时间线折叠设计

## 背景与证据

故障闭环当前把 Case 的全部 `TransitionEvent` 直接渲染在“过程时间线”中。截图中的 Case 已多次在“验证中”和“等待证据”之间往返，时间线随事件累计不断增长，进而拉长当前 Case 主栏和整个页面。用户需要滚动很远才能到达下方证据与阶段输出，页面的信息密度和定位效率持续下降。

当前实现位于 `BugCaseLifecycle.vue`：

- 模板使用 `[...detail.events].reverse()` 展示全部事件；
- 最新事件位于最上方；
- 时间线没有折叠状态、数量摘要、可视上限或内部滚动容器。

问题属于前端渐进披露和布局约束，不需要修改事件持久化、状态机或后端接口。

## 已确认需求

- 默认只显示最新 3 条时间线事件。
- 标题区显示事件总数。
- 用户可以展开查看全部事件，也可以再次收起。
- 展开后必须使用固定高度的内部滚动区域，不能重新把页面无限拉长。
- 切换 Case 后恢复默认收起状态。

## 目标

- 用户进入 Case 时立即看到最近发生的 3 个事件。
- 历史事件保持完整可访问，但不主导页面高度。
- 事件持续增加时，时间线对主页面高度的占用保持有界。
- 鼠标、触控和键盘均可操作展开/收起。
- 保持现有事件顺序、内容、状态文案和数据来源不变。

## 方案比较

### 方案 A：最近 3 条预览，展开后内部滚动（采用）

时间线标题区增加数量和展开按钮。收起时只渲染最新 3 条；展开后渲染全部事件，但列表放入固定高度滚动视口。

优点是默认保留最新上下文，同时彻底限制页面高度；交互成本低，也符合数据密集型工作台的渐进披露模式。代价是查看较早事件需要一次点击和内部滚动。

### 方案 B：原生 `<details>` 默认完全收起

实现最轻，但用户进入 Case 后看不到任何最近进展，需要额外点击，不适合需要持续观察阶段推进的故障闭环。

### 方案 C：弹窗或侧边抽屉查看完整时间线

可提供更大阅读空间，但会增加遮罩、焦点管理和关闭操作。时间线是当前 Case 的辅助上下文，不值得引入重型浮层。

## 交互设计

### 1. 标题区

时间线标题保持“过程时间线”，旁边显示总数：

```text
过程时间线 · 共 12 条                       展开全部⌄
```

- 总数始终来自 `detail.events.length`。
- 事件数小于或等于 3 条时不显示展开按钮。
- 展开按钮文案在“展开全部”和“收起”之间切换。
- 箭头随状态改变方向，但文字始终明确表达操作，不能只靠图标。
- 按钮点击区域最小 44×44px，不使用表单 `<select>`；这里的“下拉”语义是可展开区域。

### 2. 收起状态

- 将事件复制并反转，继续保持最新事件在上。
- 只渲染反转后数组的前 3 条。
- 新事件到达时，总数和顶部预览同步更新，无需用户重新展开。
- 空事件继续显示“暂无状态事件”。

### 3. 展开状态

- 渲染该 Case 的全部事件，仍然最新优先。
- 列表进入独立滚动容器：

```css
max-height: clamp(280px, 38vh, 520px);
overflow-y: auto;
overscroll-behavior: contain;
scrollbar-gutter: stable;
```

- 展开不会把全部事件高度重新计入页面布局。
- 收起后立即恢复最近 3 条预览，内部滚动位置无需持久化。
- 不添加分页、加载更多或事件过滤。

### 4. Case 切换与数据更新

组件维护一个本地布尔状态 `timelineExpanded`，默认值为 `false`。

- 监听 `detail.case.id`；Case ID 变化时设置为 `false`。
- 同一 Case 新增事件时保持用户当前的展开/收起选择。
- 当事件数从大于 3 下降到小于或等于 3 时，按钮消失，视图按收起内容展示。
- 不把折叠状态写入 SQLite、localStorage 或父页面状态。

## 可访问性

- 使用真实 `<button type="button">` 控制展开，而不是可点击的 `div`。
- 按钮设置 `aria-expanded="true|false"` 和 `aria-controls="case-timeline-events"`。
- 被控制的列表区域具有稳定 ID 和可读 `aria-label`。
- 按钮具有可见 `:focus-visible` 焦点环。
- 展开状态通过文字、箭头和 `aria-expanded` 同时表达，不只依赖颜色。
- 保持 DOM 顺序和视觉顺序一致；不引入键盘陷阱。
- 箭头旋转使用 150–200ms 的轻量过渡，并在 `prefers-reduced-motion: reduce` 下关闭。

## 视觉与响应式

- 标题和控制按钮使用弹性布局，允许标题、数量和按钮在窄屏自然换行。
- 展开按钮沿用当前工作台的紧凑次要操作风格，不与当前阶段主按钮争夺视觉层级。
- 事件圆点、事件内容、状态变化和时间信息保持现有样式。
- 375px、768px、1024px、1440px 下不得产生页面横向滚动。
- 长事件类型和时间文本继续允许安全换行。

## 组件与数据边界

改动限定在：

- `web/src/components/BugCaseLifecycle.vue`
- `web/src/components/BugCaseLifecycle.test.ts`

`BugCaseLifecycle.vue` 负责：

- 本地展开状态；
- 最新优先的事件数组；
- 收起时最近 3 条的可见数组；
- Case 切换时重置；
- 标题、按钮和内部滚动样式。

不创建新的全局状态或子组件；当前时间线逻辑足够小，留在生命周期组件内更容易维护。

## 测试策略

先写失败测试，再修改生产代码：

1. 构造 6 条事件，验证默认只渲染最新 3 条，顺序为最新到最旧，并显示“共 6 条”。
2. 验证超过 3 条时出现“展开全部”按钮，具有 `aria-expanded="false"` 和正确的 `aria-controls`。
3. 点击展开后验证 6 条全部出现、按钮变成“收起”、`aria-expanded="true"`。
4. 再次点击收起后恢复最新 3 条。
5. 验证 0–3 条事件不显示展开按钮，空状态保持。
6. 展开 Case A 后切换到 Case B，验证自动恢复收起状态并只显示 Case B 最近 3 条。
7. 同一 Case 新增事件时保持当前展开状态，并更新总数和预览内容。
8. 源码样式测试锁定 `clamp(280px, 38vh, 520px)`、内部纵向滚动、稳定滚动条、overscroll containment、44px 点击高度、焦点样式和 reduced-motion。
9. 运行 `BugCaseLifecycle` 聚焦测试、故障闭环页面相关测试、Web 全量测试、`vue-tsc --noEmit` 和生产构建。

## 非目标

- 不翻译或改写 `event_type`。
- 不改变事件写入、排序真源、状态流转或审计数据。
- 不增加分页、搜索、过滤、按阶段分组或事件删除。
- 不持久化展开状态。
- 不把时间线移到弹窗、抽屉或 Case 详情下方。
- 不调整当前阶段操作卡、进度条、Case 列表或下方阶段输出。

## 验收标准

- 事件超过 3 条时，默认只显示最新 3 条和准确总数。
- 用户可以通过“展开全部 / 收起”查看完整历史并恢复预览。
- 完整时间线始终限制在 `clamp(280px, 38vh, 520px)` 的内部滚动区域，不再拉长整个页面。
- Case 切换后默认收起；同一 Case 更新时保留当前选择。
- 事件不超过 3 条时没有多余按钮。
- 展开按钮可通过鼠标、触控和键盘操作，并具有正确无障碍属性和焦点样式。
- 原有事件内容、顺序、空状态和故障闭环业务行为保持不变。
- 相关测试、TypeScript 检查和生产构建全部通过。
