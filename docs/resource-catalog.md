# 创建向导资源目录与部署就绪度

本文记录当前已经落地的资源目录模型、兼容边界和验收要求。历史设计过程由 Git 历史和 `docs/decisions.md` 保存，本文只描述当前事实。

## 目标

创建向导的代码仓库、配置源、数据层和可观测性不是四组彼此独立的表单，而是在描述同一套系统资源。改造目标是让用户先建立代码和服务身份，再自动发现运行时资源，最后只处理冲突和缺口。

当前 Schema 0.2 同时支持正式 `resource_catalog` 和旧 YAML 派生兼容。新向导生成正式目录；旧 0.1 YAML 导入时自动派生目录，不要求用户手工迁移。

## 核心模型

资源目录按以下关系组织：

```text
代码仓库 / monorepo 子目录
  -> 业务服务或运行时工作负载
    -> 环境
      -> 分支
      -> 配置源实例与资源定位
      -> 数据组件
      -> K8s workload
      -> 日志 / 指标 / Trace 身份
```

身份必须拆开：

- `repository`：源码身份，用 URL、本地路径和 sub-path 锚定。
- `service`：配置中心和数据层使用的业务服务身份。
- `workload`：K8s、日志和部署观测使用的运行时身份；前端仓库也可以有 workload。
- `source instance`：配置源实例，不再假设 type 就是唯一 ID。

## 部署等级

向导最终生成能力覆盖矩阵，并给出三个等级：

- 基础：代码仓库完整，可做静态排障。
- 标准：代码仓库完整，且至少具备配置源或日志/K8s 运行时能力。
- 完整：配置、数据层、日志、指标、Trace 和精确 workload 映射均可用。

缺少可选能力允许降级部署，但必须明确显示影响；缺少会造成错误产物的字段必须阻止下一步。

## 当前正确性和就绪度规则

1. 仓库步骤要求扫描成功，或由用户明确手工确认技术栈；避免前端放行、后端最终校验失败。
2. 没有配置源时必须显式选择 `none`，不能把漏填静默转换为 `none`。
3. 每个配置源预读、namespace 和 dataId 状态按 source instance 隔离；同类型多个 Nacos/Apollo/Consul 实例可以并存。
4. 数据层允许手工补录自动扫描未发现的组件，并保留来源标识。
5. 可观测性启用后必须具备完整连接字段和成功探测；允许用户显式选择不启用可观测性。
6. K8s runtime 复用配置源 Kuboard/one2all 连接时，生成 YAML 也复用同一连接，避免界面可读但部署产物缺 endpoint。
7. 在可观测性步骤展示统一覆盖矩阵和部署等级。

## 当前资源目录（Schema 0.2）

- 新增顶层 `resource_catalog`，正式声明 service、workload、repository 以及逐环境的配置源、数据层和 workload 引用。
- 路由模板优先按 `(env, service)` 读取 `config_sources`，旧 `repos[].config_source` 仅作为兼容回退。
- `infrastructure.data_stores[]` 新增稳定 `id`；旧 YAML 自动按 `type`、`type-2` 规则迁移。
- 向导允许同仓多服务分属不同源，也允许同一服务在不同环境选择不同源。
- 前端/移动端 workload 可独立存在，不要求伪造配置中心 service。
- 常规 YAML、页面预览、复制内容和浏览器草稿不写真实 secret；桌面端保存到系统钥匙串，部署时从内存/钥匙串注入。
- 用户显式选择“导出可部署配置”时，Studio 从当前向导状态或系统钥匙串恢复真实凭据并生成独立敏感文件；仍有占位符就拒绝导出。该文件可直接导入部署，但必须通过受控渠道传输且禁止提交版本库。
- 旧 0.1 YAML 自动派生目录并继续使用，无需手工迁移。

## 当前边界

- Nacos/Apollo/Consul 已以 instance ID 管理凭据、预读缓存、namespace、dataId、服务归属和部署注入；同类型远程实例不会互相覆盖。Kuboard/one2all 当前各只允许一个连接实例。
- 数据层人工补录和 YAML 导入保留稳定 instance ID，探测状态按 `(instance, env, service)` 管理；自动扫描仍会先按连接类型识别，若同一配置文件声明多个同类型连接而无法区分身份，需要用户在界面补录第二实例。
- `one2all` 连接仍是系统级单 endpoint，向导不提供同一机器人多个 one2all endpoint；这是上游连接契约限制，不以相同 URL 复制实例。

## 兼容策略

- 旧 YAML 继续按现有 importer 读取。
- 可部署配置允许导入真实 secret 并直接部署；普通脱敏 YAML 在目标机器缺少对应钥匙串值时仍需补充凭据。
- 旧多源 YAML 可导入，并按 `config_centers[].id` 恢复实例级状态；缺失 ID 的旧单源按类型恢复后重新生成 0.2 YAML。
- 数据层手动补录仍写入现有 data store endpoint 结构，不引入新的 runtime 依赖。

## 验收标准

- 用户不能生成“向导显示完成，但后端因缺 stack 失败”的 YAML。
- 用户不能误把未选择配置源解释为系统明确无配置中心。
- 任何界面中成功复用的 K8s 连接必须进入最终 YAML。
- 自动扫描未识别时，可以手工增加数据组件并完成 probe。
- 覆盖矩阵能够逐环境、逐服务说明代码、配置、数据、workload、日志和 Trace 的精度或缺口。
- 所有新增路径有纯前端测试；生成行为有 YAML 单测；Go schema 保持兼容并通过全量测试。
