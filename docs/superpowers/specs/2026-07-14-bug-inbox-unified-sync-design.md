# Bug 收件箱统一同步入口设计

## 背景

Bug 工单页当前把“从平台同步”放在平台配置区，把“刷新列表”放在收件箱。两个入口分别执行远端同步和本地读取，但用户的实际目标是看到所有已启用平台中最新、指派给自己的 Bug；保留两个入口增加了判断成本，也让单平台同步与多平台配置不一致。

本设计取代 `2026-07-13-bug-inbox-sync-refresh-actions-design.md` 中“保留两个独立入口”的交互结论，不修改后台定时同步和按 ID 拉取的语义。

## 目标

- 删除平台配置区中的手动“从平台同步”按钮。
- 将收件箱“刷新列表”替换为唯一的“同步我的 Bug”入口。
- 一次操作尝试同步所有已启用平台，不能只同步当前选中的平台。
- 单个平台失败不阻断其余平台，结束后统一刷新本地 Bug 列表并给出汇总反馈。
- 重新整理“同步与接入”布局，使自动同步、指定 Bug 和 Hook URL 的职责清晰。

## 方案

采用前端统一编排，复用现有 `listBugPlatforms`、`syncBugPlatform` 和 `listBugs` 接口，不新增后端批量同步 API。

点击“同步我的 Bug”后：

1. 从已加载的平台配置中筛选 `enabled === true` 的平台。
2. 按平台列表顺序逐个调用 `syncBugPlatform(platform.id)`；避免多个同步过程并发写同一本地工单文件，且不能因某个平台失败提前结束。
3. 汇总成功平台数、失败平台名称以及新增/更新总数。
4. 无论部分平台是否失败，最后调用一次工单列表加载流程，使成功平台的结果立即可见。
5. 全部成功时显示成功摘要；部分失败时显示部分成功摘要并列出失败平台；全部失败时显示失败摘要。
6. 没有已启用平台时不发起同步，提示用户先启用 Bug 平台；现有本地列表保持不变。

同步期间按钮禁用，文案改为“同步中…”，图标使用现有旋转状态。列表加载态与平台同步态共同决定按钮是否可操作，避免重复提交。

## 页面布局

“同步与接入”区域删除手动同步按钮后保留三层：

- 第一行：后台定时同步开关、间隔输入。
- 第二行：带可见标签的“指定 Bug”输入框和“拉取指定 Bug”按钮，输入区占据剩余宽度。
- 第三行：Hook URL、地址和复制按钮，长地址允许换行但不挤压复制操作。

收件箱标题区右侧的原“刷新列表”按钮改为“同步我的 Bug”，继续使用紧凑次要按钮样式，不与“保存配置”争夺主操作层级。移动端三层布局纵向排列，按钮保持至少 44px 触控高度。

## 错误与反馈

- 单个平台失败时记录平台名称和标准化错误文本，继续同步其他平台。
- 列表最终刷新失败时单独提示“读取 Bug 工单”错误，不能把远端同步成功伪装成整体失败。
- 部分成功反馈包含成功平台数、失败平台数和新增/更新总数；失败详情通过错误提示展示，不在页面永久堆积。
- 同步过程不自动修改平台启用状态、登录状态或定时同步配置。

## 测试

- 页面不再渲染 `sync-platform` 操作。
- 唯一同步按钮对每个已启用平台调用一次 `syncBugPlatform`，忽略停用平台。
- 一个平台失败时仍同步后续平台，并在结束后只刷新一次本地列表。
- 没有已启用平台时不调用同步接口，并显示引导提示。
- 同步期间按钮禁用、显示“同步中…”并保留旋转图标反馈。
- “同步与接入”区域不再包含手动同步按钮，指定 Bug 与 Hook URL 布局契约仍适配桌面和移动端。
- 运行 Bug 工单页聚焦测试、全量 Web 测试、生产构建、Go 全量测试和 lint。

## 非目标

- 不改变后台定时同步调度。
- 不改变“拉取指定 Bug”的当前平台作用域。
- 不新增服务端批量同步接口、数据库字段或并发队列。
- 不删除底层单平台同步能力，因为后台轮询和统一入口仍需复用。
