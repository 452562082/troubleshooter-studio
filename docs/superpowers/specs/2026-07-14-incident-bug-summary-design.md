# 故障闭环 Bug 摘要精简设计

## 背景

“故障闭环”页面当前在右上角提供“刷新 Bug”按钮。该按钮仅重新读取本地已同步工单，与“Bug 工单”页面的“同步我的 Bug”入口语义接近，容易让用户误以为两处都会从外部平台同步数据。

页面中间的工单摘要复用了通用 `BugTicketDetail` 的 summary 模式，展示系统、环境和服务，但这些字段不是用户在选择故障 Case 时最关心的信息，也挤占了 Case 流程的首屏空间。

## 目标

- 将外部 Bug 同步入口统一保留在“Bug 工单”页面。
- 故障闭环进入页面时仍自动读取本地已同步工单，保持路由选中和 Case 恢复能力。
- 将所选 Bug 精简为适合故障闭环决策的摘要信息。
- 明确展示标题、Bug 等级、创建时间、更新时间和 Bug 状态。
- 保持现有 Case、机器人选择、重置和阶段流转行为不变。
- 在 375px、768px、1024px、1440px 下保持可读且不产生横向滚动。

## 设计

### 1. 移除故障闭环的手动刷新入口

删除页面标题区右侧的“刷新 Bug”按钮及仅由该按钮使用的 `refreshTickets` 方法。

不删除页面挂载时的 `tickets.load()`：故障闭环仍需从本地存储读取 Bug 列表，并根据路由参数选中工单、匹配机器人及恢复已有 Case。页面不再提供显式同步或刷新动作；用户需要获取平台最新数据时，从“Bug 工单”页面执行“同步我的 Bug”。

标题说明调整为能够表达职责边界：Bug 数据由“Bug 工单”统一同步，本页负责选择工单并推进故障闭环。

### 2. 使用独立的故障闭环 Bug 摘要组件

新增面向故障闭环场景的纯展示组件，避免修改 `BugTicketDetail` 的通用 summary/full 行为，也避免影响“Bug 工单”页面的完整详情展示。

摘要视觉顺序为：

1. 来源与 Bug ID，例如“禅道 #840”，作为轻量追溯标识；
2. Bug 标题；
3. 四项核心元数据：Bug 等级、创建时间、更新时间、Bug 状态。

Bug 等级同时展示严重程度和优先级，格式为 `S3 · P3`。字段缺失时只展示存在的值；两者均缺失时显示 `-`。

Bug 状态使用文字状态徽标，同时在元数据中保留明确的“Bug 状态”标签，不能只依赖颜色表达。创建时间和更新时间使用现有本地时间格式，缺失时显示 `-`。

不展示系统、环境、服务、描述、复现步骤、附件或其他完整工单内容。未选中 Bug 时展示简洁空状态，引导用户从左侧选择工单。

### 3. 页面布局

桌面端继续保持三列选择区：

```text
[Bug 收件箱]  [所选 Bug 摘要 / Case 启动入口]  [排障机器人]
```

中间摘要使用紧凑信息网格：宽屏为两列，字段内容允许安全换行；窄屏变为单列。标题、来源和时间字段均设置 `min-width: 0` 与必要的断词规则，避免长标题或异常字段撑破面板。

现有独立启动卡片、流程提示、Case 列表和生命周期面板保持原位置与行为。移除顶部刷新按钮后，标题区不再为右侧操作预留空间。

### 4. 组件边界

#### 新摘要组件

- 只接收 `BugRecord` 或空值。
- 负责来源、标题、等级、时间和状态的格式化与展示。
- 不读取数据、不触发同步、不管理 Case。
- 使用语义化标题和描述列表，便于测试与辅助技术读取。

#### `IncidentWorkbenchPage.vue`

- 继续负责本地工单加载、工单选择、机器人匹配和 Case 生命周期。
- 使用新摘要组件替换 `BugTicketDetail mode="summary"`。
- 删除手动刷新入口，但保留初始化加载和路由同步逻辑。

#### `BugTicketDetail.vue`

- 不改变 full 模式及 Bug 工单页面行为。
- 本次不承担故障闭环摘要展示，降低不同页面间的展示耦合。

## 可访问性与响应式

- 状态徽标同时提供可读文字，不以颜色作为唯一信息。
- 时间使用 `<time>` 元素，并保留原始时间值作为 `datetime`（有效时）。
- 元数据标签与值使用语义化 `<dl>/<dt>/<dd>`。
- 长标题、来源和字段值允许安全换行，不截断关键状态信息。
- 1024px 以下沿用现有两列/机器人整行规则，700px 以下变为单列。
- 不新增 hover 才可发现的操作，不改变现有键盘导航路径。

## 测试策略

先写失败测试，再实现：

1. 新摘要组件测试标题、来源、`S3 · P3` 等级、创建时间、更新时间和状态。
2. 测试等级字段在仅有严重程度、仅有优先级、两者均缺失时的降级展示。
3. 测试空 Bug 状态以及详情字段不出现在摘要中。
4. 页面测试确认不再渲染“刷新 Bug”按钮，且不存在对应点击行为。
5. 页面测试确认初始化仍调用本地 Bug 加载，路由选中和已有 Case 恢复不受影响。
6. 页面测试确认使用新摘要组件，系统、环境、服务及完整详情不再出现在选择摘要区域。
7. 运行相关组件与页面聚焦测试、全量前端测试、TypeScript 检查和生产构建。

## 非目标

- 不修改“Bug 工单”页面的同步算法、同步按钮或平台配置。
- 不删除故障闭环初始化时的本地 Bug 加载。
- 不改变 Bug 本地存储格式或后端 API。
- 不改变 Case 创建、重置、恢复、验证、排障、修复、合并、部署或回归流程。
- 不修改“Bug 工单”页面的完整详情展示。

## 验收标准

- 故障闭环页面不再出现“刷新 Bug”按钮。
- 进入故障闭环后仍能看到本地已同步工单，并能根据路由选中对应 Bug。
- 所选 Bug 仅展示来源/ID、标题、`严重程度 · 优先级`、创建时间、更新时间和状态。
- 系统、环境、服务及具体工单详情不再占用故障闭环摘要区。
- Case 启动、机器人选择、已有 Case 恢复和后续阶段流转行为保持不变。
- 桌面、中屏和移动端无内容越界或横向滚动。
- 相关测试、TypeScript 检查和生产构建全部通过。
