# 故障 Case 详情下置与阶段输出滚动设计

## 背景

故障闭环的 Case 生命周期区域当前采用三列同一 Grid 行：左侧“故障 Cases”、中间当前 Case、右侧“验证证据 / 根因结论 / 阶段输出 / 代码变更 / 授权 / 部署观察”。当 Agent 阶段输出很长时，右侧详情列会把整行高度撑大，导致左侧和中间内容下方出现大面积空白。用户需要持续向下滚动，Case 列表和当前 Case 的信息层级也被长日志削弱。

## 目标

- 桌面端把“故障 Cases”和当前 Case 保留在上方主工作区。
- 将原右侧“Case 证据与详情”整体移动到上方主工作区下方并横跨全宽。
- 让“阶段输出”独占详情网格整行，其内容在卡片内部独立纵向滚动。
- 阶段输出可视高度随窗口高度同比例变化，同时避免极端小窗或大屏下过小、过大。
- 保持 Case 选择、刷新、主操作、重置、详情跳转和所有数据内容不变。
- 在 375px、768px、1024px、1440px 下不产生页面横向滚动。

## 根因

`BugCaseLifecycle.vue` 的 `.case-lifecycle` 当前使用三列：

```text
[故障 Cases] [当前 Case] [证据与详情]
```

三个区域处于同一 Grid 行，行高由最高的右侧详情决定。`BugCaseArtifacts.vue` 又把所有阶段的 `output_json` 和历史事件完整展开，因此长输出不仅增加页面总高度，还会同步拉高左、中的视觉行高。

问题不是文本断行，而是主信息和长详情被放在同一个横向 Grid 行中。

## 方案比较

### 方案 A：上方双列、下方全宽详情（采用）

生命周期根布局改为两列，详情列通过 `grid-column: 1 / -1` 放到下一行。阶段输出卡片内部增加响应式滚动视口。

优点是直接修复 Grid 行高根因，保持现有组件和数据流，桌面与窄屏行为清晰。代价是详情需要自然滚动到主工作区下方查看。

### 方案 B：保留三列，整个右侧详情列内部滚动

可以限制右列高度，但验证证据、根因、代码变更等信息也会被塞进同一个滚动容器，难以扫读，且不符合“最右边整体放到下面”的要求。

### 方案 C：允许用户拖拽调整详情高度

交互自由度高，但需要拖拽手柄、持久化和键盘替代操作，明显超出本次布局问题的必要范围。

## 设计

### 1. 生命周期上方主工作区

桌面端 `.case-lifecycle` 改为两列：

```text
[故障 Cases] [当前 Case / 阶段 / 操作 / 时间线]
```

- 左列使用紧凑的 Cases 宽度，右列消费剩余空间。
- 两列均保持 `min-width: 0`，长 Case ID、状态和阶段内容不能撑破容器。
- 当前 Case 的阶段进度、状态操作卡和时间线不改变 DOM 顺序和行为。
- 两列的行高只由上方主工作区内容决定，不再受阶段输出长度影响。

### 2. 下方 Case 证据与详情

`.case-detail-column` 保留为 `BugCaseLifecycle` 的第三个语义区域，但在 Grid 中位于第二行并设置 `grid-column: 1 / -1`，横跨完整可用宽度。

`BugCaseArtifacts` 继续负责：

- 重置关系；
- 验证证据；
- 根因结论；
- 阶段输出；
- 代码变更与测试；
- 授权记录；
- 部署观察。

桌面端详情内部使用两列响应式网格，短卡片可以并排；“阶段输出”卡片增加明确类名并独占整行。窄屏下所有卡片改为单列。

### 3. 阶段输出独立滚动

“阶段输出”的标题保持在卡片顶部，不进入滚动区域。标题下面新增 `.attempt-output-scroll`，包裹空状态、普通 attempt 和历史 projection。

桌面和中屏使用：

```css
height: clamp(320px, 45vh, 640px);
overflow-y: auto;
scrollbar-gutter: stable;
```

效果：

- 窗口高度增加时，输出视口按 `45vh` 同比例增大；
- 小窗口不低于 320px，保证内容仍可操作；
- 超大窗口不超过 640px，避免日志占满整页；
- 长输出只在该区域内部滚动，不再无限拉长页面。

移动端使用：

```css
height: clamp(280px, 42vh, 480px);
```

不提供“展开全部”、折叠状态或用户拖拽高度，不新增持久化数据。

### 4. 滚动交互与可访问性

- `.attempt-output-scroll` 使用 `role="region"` 和可读 `aria-label="阶段输出内容"`。
- 添加 `tabindex="0"`，键盘用户聚焦后可用方向键、Page Up/Down、Home/End 浏览长内容。
- 保持可见焦点环，不能只靠滚动条表达区域可交互。
- 保留浏览器原生滚动条；触控设备支持原生惯性滑动。
- `overscroll-behavior: contain`，滚动到边界时避免意外带动整个页面。
- `pre`、`code` 和长路径继续安全换行，不产生横向滚动。

## 组件边界

### `BugCaseLifecycle.vue`

- 只调整三块既有区域的 Grid 排列和响应式规则。
- 不移动数据所有权，不改变事件接口。
- 桌面为上方两列、下方详情整行；899px 以下上方也变为单列，详情保持在其后。

### `BugCaseArtifacts.vue`

- 保持全部详情内容和计算逻辑不变。
- 为阶段输出卡片增加稳定类名和内部滚动容器。
- 负责详情网格中阶段输出跨列、滚动高度和焦点样式。

不把详情拆到页面父组件，避免复制 `detail` 传递和 `select-case` 事件。

## 数据流和错误处理

本次没有新的状态或异步请求：

```text
IncidentWorkbenchPage
  → BugCaseLifecycle(detail, cases)
    → BugCaseArtifacts(detail)
```

Case 选择、刷新和关系跳转事件保持原样。空 Case、空证据、空阶段输出仍使用现有空状态。长输出的渲染和安全文本处理逻辑不变，只改变可视容器。

## 响应式规则

- 大于 899px：上方两列；详情位于下一行全宽；详情卡片两列；阶段输出跨两列。
- 小于等于 899px：生命周期全部单列；详情卡片单列；阶段输出高度使用移动端比例。
- 小于等于 560px：保留现有阶段进度两列、Case 标题堆叠和对话框按钮堆叠。
- 375px、768px、1024px、1440px 均不得出现页面级横向滚动。

## 测试策略

先写失败测试，再改生产代码：

1. `BugCaseLifecycle.test.ts` 验证根布局为上方两列，详情区设置 `grid-column: 1 / -1`，且 DOM 顺序为 Cases、当前 Case、下方详情。
2. 验证 899px 以下生命周期改为单列，现有 560px 阶段与对话框规则仍存在。
3. `BugCaseArtifacts.test.ts` 验证阶段输出卡片独占详情网格整行。
4. 验证 `.attempt-output-scroll` 包含普通和历史 attempt，标题位于滚动容器之外。
5. 验证滚动区域具有 `role="region"`、`aria-label`、`tabindex="0"` 和可见焦点规则。
6. 验证桌面 `clamp(320px, 45vh, 640px)`、移动端 `clamp(280px, 42vh, 480px)`、`overflow-y: auto`、稳定滚动条和 overscroll containment。
7. 运行两个组件聚焦测试、故障闭环页面测试、前端全量测试、TypeScript 检查和生产构建。

## 非目标

- 不改变 Case 状态机、按钮、授权、重置或阶段流转。
- 不删除、截断或重新解释任何阶段输出。
- 不增加折叠、展开全部、拖拽高度或高度持久化。
- 不改变 Agent 输出、后端 API、SQLite 数据或 bridge 类型。
- 不重构 `BugCaseArtifacts` 的 Markdown 和安全文本处理。

## 验收标准

- 桌面端“故障 Cases”和当前 Case 并排位于上方。
- “Case 证据与详情”位于下方并横跨完整宽度。
- 长阶段输出不再拉高上方主工作区或制造大面积空白。
- 阶段输出标题保持可见，内容区域可独立滚动。
- 输出区域随窗口高度变化：桌面 `clamp(320px, 45vh, 640px)`，移动端 `clamp(280px, 42vh, 480px)`。
- 鼠标、触控和键盘均可浏览完整输出。
- 原有 Case 选择、刷新、关系跳转、主操作、重置和所有内容保持不变。
- 相关测试、前端全量、TypeScript 检查和生产构建全部通过。
