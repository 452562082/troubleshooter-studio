# 故障闭环阶段输出结构化展示与自动滚动设计

## 背景与证据

故障闭环的“阶段输出”当前把非历史 `PhaseAttempt.output_json` 统一交给 `JSON.stringify`，再放入 `<pre>`。截图中的验证结果因此直接暴露 `verification_status`、`expected_behavior`、`gaps` 等机器字段和 JSON 标点。用户必须理解字段含义后才能找到结论、实际表现和缺失资料，多次验证结果还会连续铺开，信息层级不清楚。

`BugCaseArtifacts.vue` 已经对导入的 legacy 记录做了专门的安全文本投影，但对当前工作流仍使用通用 `displayJSON`。后端实际上为各阶段提供了稳定契约：

- 验证和回归：`verification_status`、环境、预期表现、实际表现、证据和缺口；
- 排障：`investigation_status`、环境、根因、置信度、证据和缺口；
- 修复：`fix_status`、环境、分支、变更、测试、部署说明、风险、阻塞原因和证据。

因此问题在展示层，不需要修改工作流状态、SQLite 数据或 Agent 输出协议。

## 已确认需求

- 阶段输出不能默认或按入口展示原始 JSON。
- 内容必须以用户能直接扫读的中文结果卡片展示。
- 最新阶段输出优先，历史尝试保持可访问但减少视觉占用。
- 阶段输出区域在首次打开、切换 Case 和收到新结果后自动滚动到底部。
- 自动滚动只作用于阶段输出内部，不能改变整个页面的滚动位置。

## 目标

- 首屏先回答“当前阶段得出了什么结论”。
- 把预期、实际、缺口、证据、根因、变更和测试等内容变成有标题的语义区块。
- 所有阶段名、执行状态和结果状态使用中文，不只靠颜色表达。
- 多次尝试按时间顺序呈现，最新一次默认展开，较早尝试默认折叠。
- 保持现有阶段输出独立滚动高度、键盘浏览和响应式布局。
- 对旧数据和异常字段安全降级，不重新暴露 JSON。

## 方案比较

### 方案 A：按阶段建立结构化结果视图（采用）

前端根据 `attempt.phase` 和已知输出契约生成展示模型，再渲染中文标题、状态徽标、说明区块和列表。未知或旧字段使用通用的“名称—内容”结构化兜底，但不显示 JSON 源文。

优点是符合业务语义，用户无需理解机器字段；后端契约保持不变。代价是前端需要维护少量阶段字段映射。

### 方案 B：只做通用 JSON 树形查看器

能快速消除大段 `<pre>`，也能适配未知字段，但仍以技术字段为中心，无法突出“信息不足”“还需补充”和“根因已确认”等业务结论。

### 方案 C：后端额外生成 UI 展示模型

可以统一所有客户端，但会让持久化层和 UI 文案耦合，并扩大 API、迁移和状态机测试范围。本次问题只在单个前端组件，不值得增加跨层协议。

## 展示结构

### 1. 尝试标题与状态

每次尝试显示：

- 阶段中文名：验证、排障、修复、回归验证、历史运行；
- 结果状态徽标；
- 执行状态和时间作为次要信息；
- `error_message` 存在时显示为独立错误提示，不与正文混排。

执行状态映射为“排队中、执行中、已完成、失败、已取消、已中断”。结果状态至少覆盖：

- 验证/回归：已复现、未复现、信息不足、修复已验证、仍可复现；
- 排障：根因已确认、信息不足；
- 修复：修复已推送、修复受阻、修复失败。

徽标同时包含文字、图形/边框和颜色语义，不能只靠红绿区分。

### 2. 验证与回归结果

验证和回归尝试按以下优先级展示：

1. 结果状态和环境；
2. “预期表现”；
3. “实际观察”；
4. “还需补充”列表；
5. “验证证据”列表。

缺口为空时不显示空数组；证据为空时显示“尚无有效证据”。`scenario_hash` 属于内部关联字段，不在用户界面展示。

### 3. 排障结果

排障尝试展示：

1. 根因结论；
2. 置信度，映射为高、中、低；
3. 环境；
4. 还需补充；
5. 排障证据。

信息不足时用警示语义突出缺口；根因已确认时突出根因正文。现有页面顶部“根因结论”卡片继续保留，阶段输出用于呈现对应尝试的完整上下文。

### 4. 修复结果

修复尝试展示：

1. 修复结果和环境；
2. 阻塞原因或部署说明；
3. 代码变更摘要，按仓库分组；
4. 测试结果，显示命令、结果、备注或跳过原因；
5. 分支和提交信息；
6. 风险列表和证据。

提交、分支和测试命令使用等宽文本，但不把整个对象格式化成 JSON。

### 5. 兼容与兜底

- legacy 尝试继续使用现有受限 Markdown/事件投影，不改变安全边界。
- 当前工作流出现旧版 `summary`、`conclusion`、`report` 等字符串字段时，作为“阶段结论”展示。
- 其他未知标量转换为中文化的“名称—内容”行；字符串数组转换为项目列表；对象数组转换为分组卡片。
- 空对象显示“本次暂无可展示的阶段结果”。
- 不提供“查看原始数据”、展开 JSON 或复制 JSON 的入口。
- 所有内容继续通过 Vue 文本插值输出，不使用 `v-html`。

## 多次尝试与信息层级

尝试保持后端返回的时间顺序，最新记录位于滚动区域底部：

- 最新非 legacy 尝试默认展开完整结果；
- 较早尝试使用原生 `<details>` 折叠，摘要行保留阶段、结果、时间和错误状态；
- 用户可以展开历史尝试比较差异；
- legacy 记录保持当前可读投影，不改写为新阶段契约。

折叠按钮必须可键盘操作并具有清晰焦点样式。

## 自动滚动行为

阶段输出滚动容器持有稳定 DOM 引用，并在以下时机于 Vue 完成 DOM 更新后执行 `scrollTop = scrollHeight`：

- 组件首次展示已有阶段输出；
- `detail.case.id` 变化，即切换 Case；
- attempts 新增、删除，或任一 attempt 的阶段、状态、输出、错误信息发生变化；
- 当前 Case 数据刷新后产生新的可见结果。

精确交互约束：

- 只滚动 `.attempt-output-scroll`，不调用会影响祖先的 `scrollIntoView`；
- 没有新内容时，用户可以自由向上浏览历史，不周期性强制回底；
- 新内容到达时，无论用户当前位于何处，都回到底部，满足“持续跟随最新输出”；
- 自动滚动使用即时定位，不使用长动画，避免大量输出时延迟追尾；
- 容器继续使用 `role="region"`、可读标签和 `tabindex="0"`，并增加适合动态结果的 `aria-live="polite"` 或等价的非打断式通知语义。

实现时使用稳定的内容签名或深度 watcher 覆盖原地更新，并通过 `nextTick` 等待折叠状态和结构化区块完成布局。组件卸载后不得保留 observer 或定时器。

## 视觉与响应式

- 保留阶段输出卡片独占详情网格整行。
- 保留桌面 `clamp(320px, 45vh, 640px)`、移动端 `clamp(280px, 42vh, 480px)` 的内部滚动视口。
- 结构化区块使用清晰的小标题、适度留白和浅色边界，不把每个字段做成高密度大卡片。
- 缺口使用警示色和列表；错误使用危险色；成功和已确认状态使用成功色；所有颜色同时配文字。
- 长路径、命令、提交和中英文混排继续安全换行，页面和阶段输出都不能出现横向滚动。
- 375px、768px、1024px、1440px 下均可扫读。

## 组件与数据边界

主要改动限定在 `BugCaseArtifacts.vue` 及其测试：

- 将非 legacy attempt 的 `displayJSON` 替换为纯前端展示模型和语义模板；
- 增加阶段、执行状态、结果状态、置信度和字段标题映射；
- 增加历史尝试折叠与最新尝试识别；
- 增加滚动容器引用、内容变化监听和 DOM 更新后的到底逻辑；
- `displayJSON` 仍可供“代码变更与测试”等非阶段区域使用，本次不扩大到其他卡片。

不修改 `PhaseAttempt.output_json` 类型、Wails bridge、Go 结构体、SQLite schema、Agent prompt 或工作流状态机。

## 测试策略

先写失败测试，再修改生产代码：

1. 验证 validation/regression 输出不再出现 JSON 标点和 snake_case 原字段，并正确显示状态、环境、预期、实际、缺口和证据空状态。
2. 验证 investigation 输出显示根因、置信度、缺口和证据。
3. 验证 fix 输出显示阻塞原因/部署说明、仓库变更、测试、分支、提交和风险。
4. 验证未知旧字段使用结构化兜底，界面不存在原始 JSON 查看入口。
5. 验证最新尝试展开、历史尝试折叠，阶段名和状态均为中文。
6. 验证 hostile 文本仍作为纯文本渲染，不产生 HTML、脚本或可执行链接。
7. 在挂载、切换 Case、追加 attempt 和更新 output/status/error 后，验证滚动容器的 `scrollTop` 被设为 `scrollHeight`。
8. 验证自动滚动不调用 `scrollIntoView`、不修改页面滚动位置，没有内容变化时不反复抢占用户滚动。
9. 保留既有内部滚动高度、响应式、键盘焦点和 legacy 投影测试。
10. 运行 `BugCaseArtifacts` 聚焦测试、故障闭环页面相关测试、Web 全量测试、`vue-tsc --noEmit` 和生产构建。

## 非目标

- 不修改 Agent 输出协议或要求 Agent 直接生成 HTML/Markdown 页面。
- 不改变 Case 状态、阶段流转、授权、重置、部署或回归逻辑。
- 不删除数据库中的原始结构化结果；只是不在用户界面展示 JSON。
- 不把顶部验证证据、根因结论、代码变更等卡片合并进阶段输出。
- 不增加用户可配置的自动滚动开关、暂停按钮或高度拖拽。
- 不重构整个工作台设计系统。

## 验收标准

- 截图中的验证结果以中文结构化内容呈现，不再出现原始 JSON。
- 用户无需理解字段名即可识别结果、预期表现、实际观察、缺失资料和证据。
- 排障和修复阶段同样具有对应业务语义，不退回通用 JSON。
- 最新尝试位于底部并默认展开，历史尝试可折叠查看。
- 首次打开、切换 Case 和阶段内容更新后，阶段输出内部始终定位到最新内容。
- 用户手动浏览历史且没有新结果时，滚动位置不会被无意义重置。
- 页面滚动位置不受自动滚动影响，内部滚动和响应式约束保持有效。
- 相关测试、TypeScript 检查和生产构建全部通过。
